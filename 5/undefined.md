# 학습 키워드

## TDD란

### 개념

* 테스트 주도 개발
* 소프트웨어 개발 방법론 중 하나
* 선 개발 후 테스트 방식이 아닌 선 테스트 후 개발 방식의 프로그래밍 방법

### 개발 방법

1. 테스트 케이스 작성 : 실패하는 테스트를 가장 빠르게 구현
2. 테스트 케이스를 통과하는 코드 작성 : 단위 테스트를 통과할 수 있을 정도의 최소한의 코드 작성
3. 작성한 코드 리팩토링 : 테스트 코드를 통해 다시 기능 코드를 점검

\=> 기능 테스트가 완전해질 때까지 2, 3의 과정을 반복

### 장점

* 객체 지향적인 코드 개발
*  설계 수정시간의 단축
* 유지보수(리팩토링)의 용이성
* 테스트 문서의 대체 가능

### 단점

* 생산성 저하
* 사전준비 기간

## Jest

* 테스팅 프레임워크
* 테스트 실행 환경 제공
* 테스트 결과 확인 후 리포트 생성

### 코드 보기

```js
test("테스트 설명", () => {
  expect("검증 대상").toXxx("기대 결과");
});
```

### 예시

```js
test("1 is 1", () => {
  expect(1).toBe(1);
});
```



## Describe - Context - It 패턴

* **실제 테스트 대상을 구체적으로 설명**하는 테스트 케이스 작성

### 키워드

* Describe : 테스트 대상 명시
* Context : 테스트 상황이나 조건 설명
* It : 테스트 수행 / 결과 검증&#x20;

## 단위테스트란

* 하나의 모듈을 기준으로 독립적으로 진행되는 가장 작은 단위의 테스트

### 장점

* 테스팅 시간과 비용 절감
* 새로운 기능 추가시 빠른 테스트 가능
* 리팩토링 안정성 확보
* 코드 문서

## React Testing Library

* 컴포넌트 테스트 도구

### 주요기능

* **DOM Query Functions**:
  * `getBy`, `queryBy`, `findBy` 등의 쿼리 메서드를 사용해 DOM 요소를 선택합니다.
  * 접근성 속성(`aria-label`, `role`)을 기반으로 테스트를 권장합니다.
* **Render**:
  * `render` 메서드를 사용해 컴포넌트를 DOM에 렌더링합니다.
* **User Events**:
  * `@testing-library/user-event`를 통해 사용자의 입력 및 클릭 이벤트를 시뮬레이션합니다.
* **Assertions**:
  * Jest와 함께 사용하여, `expect`를 기반으로 테스트를 작성합니다.

### 장점

* 사용자 관점의 테스트 설계
* 접근성 중심의 테스트
* 리액트 컴포넌트의 동작을 간단하고 명확하게 검증

### 단점

* DOM 외부 로직(상태 관리, API 호출 등)은 추가적으로 목(mock)을 작성
* 복잡한 상태 의존성 테스트는 추가 설정이 필요

## given - when - then 패턴

* given : 테스트 **준비** 과정
* when : 실제로 액션하는 테스트를 **실행**하는 과정
* then : 테스트 **검증** 과정

## Mocking

* 테스트하고자 하는 코드가 의존하는 function이나 class에 대해 모조품을 만들어 '일단' 돌아가게 하는 것
* 테스트하고 싶은 기능이 다른 기능과 엮어있을 경우 테스트하기 힘들어 가짜를 대처함

## Test fixture

* 테스트의 상태와 환경을 일관적으로 고정
*  중복 발생되는 행위를 고정시켜 한곳에서 관리

### 메소드&#x20;

1. @BeforeAll : 테스트 메소드 시작 전 1번 실행
2. @AfterAll : 테스트 메서드 종료 후 1번 실행
3. @BeforeEach : 각 테스트 메소드 시작 전 실행
4. @AfterEach : 각 테스트 메소드 종료 후 실행

## Service worker

* 네트워크 요청을 가로채서 네트워크 사용 가능 여부에 따라 적절한 행동을 취하고, 서버의 자산을 업데이트

### 특징

* 높은 비용의 계산을 처리할 때, 푸시 이벤트를 생성할 때
* 백그라운드 데이터 동기화.
* 다른 출처에서의 리소스 요청을 응답.
* 위치정보, 자이로 센서 등 계산에 높은 비용이 들어가는 다수의 페이지에서 함께 사용할 수 있도록 데이터 업데이트를 중앙화
* 개발 목적으로서 CoffeeScript, Less, CJS/AMD 모듈 등의 의존성 관리와 컴파일.\
  백그라운드 서비스 훅
* 특정 URL 패턴에 기반한 사용자 지정 템플릿 제공
* 성능 향상. 사진 앨범의 다음 사진 몇 장처럼, 사용자가 필요로 할 것으로 생각되는 리소스의 pre-fetching 등

## MSW(Mock Service Worker)

* 서버를 향한 실제 네트워크 요청을 가로채서(intercept) 모의 응답 (Mocked response)를 보내주는 API Mocking 라이브러리

## polyfill(폴리필)

* 오래된 브라우저에서 없는 구현을 채워주는 스크립트

## E2E(End to End) Test

* 사용자 입장에서 테스트
* 사용자에게 노출되는 부분 테스트

## Headless Chrome

* 사용자 인터페이스 없이 브라우저 실행

### 특징

* **UI 없음**: 브라우저 창이 표시되지 않아 리소스가 절약되고 더 빠르게 실행됩니다.
* **크롤링 및 스크래핑**: 동적 콘텐츠를 처리할 수 있어 자바스크립트 기반 웹사이트 크롤링에 적합합니다.
* **테스트 자동화**: E2E(End-to-End) 테스트에 사용됩니다.
* **PDF 생성 및 스크린샷**: 웹 페이지의 PDF 파일이나 스크린샷 생성이 가능합니다.

## Puppeteer

* 구글에서 만든 노드 라이브러리로 Headless Chrome 또는 Chromiunm 제어

## Playwright

* 오픈소스 웹 테스트
* 하나의 API로 Chromium, Firefox, WebKit까지 테스트 가능

## CodeceptJS

* 엔드투엔드(E2E) 테스트를 간단하고 직관적으로 작성할 수 있도록 설계된 현대적인 테스트 프레임워크
* 사용자 행동 중심의 BDD 스타일로 테스트 코드 작성
